## 核心问题解决

### 问题1：统计了历史数据（27次而不是3次）
**原因**：移除了时间过滤，导致统计了所有历史缓存记录

**解决**：
1. 在构造函数中立即设置 `executionStartTime`
2. 恢复 `parseCacheStats` 中的时间过滤
3. 只统计 `startTime` 之后的缓存操作

### 问题2：100% 缓存命中时显示异常
**原因**：当所有操作都命中缓存时，没有 AI API 调用，`ai-profile-detail.log` 为空

**解决**：
1. 添加 `hasAICalls` 检查
2. 只在有 AI 调用时显示 token 和成本信息
3. 100% 缓存命中时显示特殊提示

### 问题3：缓存命中识别不准确（重复计数）
**原因**：
- `cache found and marked as used` 是主要的缓存命中日志
- `cache hit` 通常紧跟在 `cache found and marked as used` 之后（几十毫秒内）
- 这两条日志指的是**同一次操作**，不应该重复计数
- 原代码同时统计了这两种日志，导致重复计数

**示例**：
```
[09:52:17.207] cache found and marked as used, type: plan, prompt: 在用户名输入框输入账号：sysadmin
[09:52:18.219] cache found and marked as used, type: locate, prompt: 登录名输入框
[09:52:18.252] cache hit, prompt: 登录名输入框  ← 这是重复日志！
```
- 实际缓存命中：2 次（1 个 plan + 1 个 locate）
- 错误统计：3 次（2 个 `cache found and marked as used` + 1 个 `cache hit`）

**解决**：
1. **只统计** `cache found and marked as used`（主要日志）
2. **忽略** `cache hit`（避免重复计数）
3. 直接统计 `will append cache`（缓存未命中）
4. 总调用数 = 命中数 + 未命中数

## 技术要点

1. **时间过滤的重要性**：必须在构造函数中设置 `executionStartTime`，而不是在第一次步骤执行时
2. **缓存日志的特点**：Midscene 在初始化时就会加载缓存，这些日志会立即写入
3. **100% 缓存命中的特殊性**：不会有任何 AI API 调用，需要特殊处理显示逻辑
4. **缓存命中的识别**：直接统计日志中的 `cache hit` 和 `cache found and marked as used`
